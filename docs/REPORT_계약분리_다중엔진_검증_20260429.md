### 2.5 E2E (Docker GraphDB + Postgres 원천 → `t2s_*` + AGE 물리 그래프) — **실행함**

| 단계 | 내용 |
|------|------|
| **① GraphDB** | `docker/K-AIR-GraphDB-t2s/docker-compose.yml` 로 `kair-graphdb-t2s` 기동(호스트 `127.0.0.1:15433` → PG `kair_graphdb_t2s`, init로 `ontology_graph`·`t2s_*` 존재). |
| **② 원천 DB** | 이 환경에서 **사내 운영 Postgres에 직접 연결할 수 없어**, 계획과 동일한 스키마명 **`RWIS`** 를 갖는 **최소 검증용 Postgres**를 별도 컨테이너로 기동(`postgres:15-alpine`, 호스트 `127.0.0.1:55432`, `dept`/`emp` + `fk_emp_dept`). 스크립트: `services/meta_ingest_proto/scripts/verify_source_min_rwis.sql`. |
| **③ 인제스트** | 호스트에서 `meta_ingest_proto` 실행: `SOURCE_*`→55432, `TARGET_*`→15433, `META_DB_LABEL=verify_min_rwis`, `python ingest.py run`. |
| **④ 출력** | `t2s: tables=2 columns=5 fk_pairs=1` · `AGE: Table=2 Column=5 HAS_COLUMN=5 FK_TO=1`. |

**RDB 검증 (`t2s_*`)**

- `t2s_tables`: `db=verify_min_rwis`, `schema_name=RWIS`, 테이블 `dept`/`emp`, 코멘트 `dept`에 반영.
- `t2s_columns`: `fqn`·`dtype`·`is_primary_key`·`metadata`(ordinal_position, column_default, is_unique) 정상.

**AGE 검증 (계획서 물리 계약 샘플)**

- `:Table`: `name`, `schema`, `datasource`, `db_exists=true` 확인.
- `:Column`: `ordinal_position`, `is_primary_key`, `fqn` 접두 `verify_min_rwis.RWIS` 일치.
- `FK_TO`: `constraint`=`fk_emp_dept`, `position`=`1` (단일 컬럼 쌍).
- `HAS_COLUMN`: `_meta_ingest` Table 기준 관계 수 **5**.

상세 쿼리: `services/meta_ingest_proto/scripts/verify_age_queries.sql` (psql로 stdin 파이프 실행).

**운영 Postgres로 동일 검증을 할 때**: `SOURCE_PG_*` 만 실제 호스트/포트/계정/DB/스키마로 바꾸고, GraphDB는 위와 동일하게 두면 된다. 원천 컨테이너는 필요 없음.

### 2.6 속성 키·재적재 (`verify_5tables_rwis`, 전용 GraphDB)

**목적**: `_physical_vertex_id` 계약 적용 후 **`t2s_*`를 비우고 다시 적재**하고, **`:Table` 5개·테이블 `dept`·`emp`의 `:Column`**에 대해 AGE `keys()` 결과를 스냅샷한다.

| 단계 | 내용 |
|------|------|
| **원천 DDL** | `services/meta_ingest_proto/scripts/verify_source_5tables_rwis.sql` → 검증 Postgres(예: `127.0.0.1:55432`, DB `rwis`, 스키마 `RWIS`). 테이블 `asset`, `dept`, `emp`, `region`, `site`, FK 3건. |
| **t2s 삭제** | `services/meta_ingest_proto/scripts/wipe_t2s_verify_5tables_rwis.sql` → 대상 GraphDB에서 `db='verify_5tables_rwis'` 행만 `DELETE` (컬럼·FK는 CASCADE). AGE 물리층은 다음 `ingest`의 `build_physical_meta_refresh`가 `_meta_ingest` 정점을 먼저 `DETACH DELETE` 하므로 별도 삭제 불필요. |
| **인제스트** | `META_DB_LABEL=verify_5tables_rwis`, `SOURCE_PG_*`→55432, `TARGET_PG_*`→**15434**(`kair-graphdb-meta-e2e`) 기준으로 `python ingest.py run`. |
| **AGE keys** | `services/meta_ingest_proto/scripts/verify_age_property_keys.sql` 를 GraphDB에 대해 실행. |

**인제스트 출력 (2026-04-29 재실행)**

## 4. 계획 대비 남는 여지 (명시)

- Oracle/MySQL/Tibero **실추출·실DB 스모크**는 계획상 골격 단계로 남김.  
- `analyze`/ROBO 경로의 `Analyzer_Table` 등 **기존 다른 라벨**과의 통합은 본 작업 범위에 포함하지 않음(물리 계약은 `:Table`/`:Column` 인제스트 경로 기준).

## 5. 최종 검토 요약

- **신규 GraphDB(t2s) Docker** 기동 후 **호스트 기반 `meta_ingest_proto` → `t2s_*` upsert + `ontology_graph` 물리 노드/관계** 경로가 **끝까지 통과**함.
